# 后台与小程序共用功能模块清单

> 生成日期：2026-06-16  
> 适用范围：奉飞飞后台管理原型、微信小程序原型、两端共用数据与后续接口设计。  
> 依据文件：`apps/admin/docs/project-page-function-list-v1.md`、`apps/admin/docs/features/*.md`、`apps/miniapp/src/router/routes.js`、`apps/miniapp/src/pages/*.js`、`apps/miniapp/src/state/appState.js`、`shared/data/catalog.js`。

## 一、整体串联口径

本项目当前是纯静态原型，没有真实后端。后台、小程序和共用数据的关系可以按“后台配置/处理、用户端提交/查看、飞手端承接/反馈”三条线理解。

| 端 | 主要职责 | 当前数据位置 | 后续接口设计含义 |
|---|---|---|---|
| 后台管理端 | 配置首页、维护商品、查看用户、处理订单、审核飞手、查看报备/发票/反馈/运营商申请、发布任务 | `apps/admin/js/modules/*.js` 内存 mock | 作为管理端 CRUD、审核、派单、配置和统计入口 |
| 微信小程序普通用户端 | 浏览首页和商品、下单填写需求、管理订单、提交培训/发票/反馈/运营商申请、查看关于我们 | `apps/miniapp/src/pages/*.js`、`state/appState.js` | 作为用户提交业务数据和查看处理进度入口 |
| 微信小程序飞手端 | 提交飞手认证、查看任务大厅、提交任务意愿、查看分配订单、提交飞行报备 | `apps/miniapp/src/pages/forms.js`、`static.js`、`state/appState.js` | 与普通用户共用账号，认证通过后开放飞手工作台能力 |
| 共用资源层 | 分类、首页导航、商品、订单状态、个人中心服务入口、共用图片 | `shared/data/catalog.js`、`shared/assets` | 后续建议沉淀为基础配置、枚举字典和素材表 |

## 三、端到端字段映射

### 1. 小程序配置

| 字段/对象 | 后台来源 | 小程序消费 | 说明 |
|---|---|---|---|
| 轮播素材 | `state.media[]`：`id`、`type`、`name`、`url`、`poster`、`enabled` | 首页顶部轮播/品牌视觉区 | 当前后台支持 1 个视频固定首位、最多 5 张图片；小程序首页当前以静态品牌 hero 展示，后续应接入同一轮播配置 |

| 推荐商品 | 后台商品列表 | 小程序首页热销商品 | 小程序当前展示 `hoistingProducts`，后续应按后台商品上架状态、销量和分类筛选 |

### 2. 商品与服务目录

| 字段/对象 | 后台维护 | 小程序展示/提交 | 进入订单后的处理 |
|---|---|---|---|
| 分类 | 分类名称、说明、排序、启用状态、商品数 | 名称、说明、分类商品 | 作为商品筛选和首页/服务入口基础配置 |
| 商品基础信息 | 商品编号、名称、分类、上架状态、商品图、介绍、评价 | 商品封面、名称、分类、起售价、详情、评价 | 下单时保存商品/服务名称、规格、价格、业务属性快照 |
| 规格 | 规格名称、价格、说明 | 选择规格、展示价格 | 生成订单明细：规格名、单价、数量、实付款 |
| 业务属性 | 是否在线支付、是否需要飞手 | 影响支付入口和订单流转提示；下单字段由商品需求字段决定 | 订单状态按在线支付和飞手需求组合生成 |
| 商品需求字段 | 每个商品单独维护字段名称、类型、必填、提示文案、选项、排序 | 确认订单页按当前商品字段动态展示需求字段 | 订单生成时保存 `requirementSnapshot`，历史订单不受商品字段调整影响 |
| 商品图片 | 最多 9 张，第一张为封面 | 商品详情轮播/列表封面 | 当前原型只展示占位或共享图片 |

### 3. 订单履约

| 字段/对象 | 小程序产生/展示 | 后台接收/处理 | 飞手端消费 | 说明 |
|---|---|---|---|---|
| 订单标识 | `orderNo` | `id` | `orderNo` | 后续接口建议统一为 `orderNo` 或 `orderId`，避免两套命名 |
| 用户信息 | 下单用户、联系人、联系电话 | 订单用户、联系手机号 | 用户、联系手机号 | 联系人和联系方式默认必填，列表可脱敏，详情按权限展示 |
| 商品快照 | 商品名称、规格、价格、服务类型 | 商品/服务、金额、在线支付、需要飞手 | 分类、商品、金额、业务属性快照 | 商品后续修改不影响历史订单 |
| 需求信息快照 | 用户按商品字段填写的动态字段，默认包含联系人、联系方式 | 后台订单详情按 JSON 快照展示完整需求 | 飞手端查看履约所需字段 | 商品字段修改不影响历史订单 |
| 支付信息 | 支付结果、实付款 | 金额、在线支付状态 | 金额 | 当前为模拟支付 |
| 派单信息 | 用户查看订单状态/进度 | 分配飞手、调整飞手、查看履约状态 | 查看分配订单、完成订单 | 后台待派单确认后进入待服务；待服务调整飞手不改变订单状态 |
| 服务通知 | 订单状态变更通知 | 由订单状态变化触发 | 无 | 小程序 `serviceNotifications[]` 当前为 mock |

订单状态默认规则：

| 支付方式 | 是否需要飞手 | 状态链路 |
|---|---|---|
| 在线支付 | 需要飞手 | 订单生成 -> 待付款 -> 待派单 -> 待服务 -> 待评价 -> 已完成 |
| 在线支付 | 无需飞手 | 订单生成 -> 待付款 -> 待交付 -> 待评价 -> 已完成 |
| 非在线支付 | 需要飞手 | 订单生成 -> 待派单 -> 待服务 -> 待评价 -> 已完成 |
| 非在线支付 | 无需飞手 | 订单生成 -> 待交付 -> 待评价 -> 已完成 |

## 四、建议后端共用数据模型

以下不是当前原型代码结构，而是从两端字段串联后建议沉淀的接口/表模型。

| 模型 | 关键字段 | 服务对象 |
|---|---|---|
| `home_media` | `id`、`type`、`name`、`url`、`poster`、`sort`、`enabled` | 后台轮播配置、小程序首页 |
| `home_nav_item` | `id`、`size`、`image`、`jumpType`、`link`、`sort`、`enabled` | 后台导航配置、小程序首页 |
| `category` | `id`、`name`、`description`、`sort`、`enabled` | 商品后台、小程序分类 |
| `product` | `id`、`code`、`name`、`categoryId`、`images[]`、`intro`、`status`、`onlinePay`、`needPilot`、`requirementFields[]` | 商品后台、小程序商品 |
| `product_spec` | `id`、`productId`、`name`、`price`、`desc`、`sort` | 商品详情、确认订单 |
| `order` | `orderNo`、`userId`、`snapshot`、`amount`、`onlinePay`、`needPilot`、`requirementSnapshot`、`status`、`createdAt` | 用户订单、后台订单、飞手订单 |
| `order_assignment` | `orderNo`、`pilotId`、`assignedAt`、`personalStatus`、`progress[]` | 后台派单、飞手任务 |
| `user_profile` | `id`、`nickname`、`phone`、`gender`、`birthday`、`region`、`role` | 用户管理、小程序我的 |

## 五、跨端交互主流程

### 1. 商品下单到飞手履约

1. 后台维护分类、商品、规格、业务属性和商品需求字段。
2. 小程序首页/分类/商品列表展示可用商品。
3. 用户进入商品详情，选择规格，填写默认联系人、联系方式及当前商品表单字段，提交订单。
4. 系统按商品业务属性生成订单状态链路，并保存商品快照与需求信息快照。
5. 后台订单列表接收订单；需要飞手的订单在“待派单”阶段分配飞手。
6. 飞手在任务大厅“分配订单”中查看需求信息快照和相关附件。
7. 飞手完成服务后更新个人履约状态；用户订单详情查看进度，后续进入评价和开票。

### 2. 飞手入驻到承接任务

1. 用户在小程序提交飞手加入申请。
2. 个人主体只提交个人、证照和设备资料；公司主体额外选择或创建公司资料。
3. 后台飞手入驻申请列表查看并审核。
4. 审核通过后，用户角色变为认证飞手，小程序展示飞手工作台入口。
5. 飞手可查看后台发布的任务需求，提交参与意愿。
6. 后台查看意愿名单，并按后续业务规则选择飞手或另行派单。

### 3. 飞行报备

1. 认证飞手在小程序提交报备。
2. 后台飞行报备列表接收待确认记录。
3. 管理员查看完整报备信息和飞行器信息。
4. 管理员确认报备；若接入第三方，再执行推送。
5. 小程序历史记录展示待确认/已确认状态。

### 4. 发票申请

1. 用户在小程序开票中心选择已完成且未开票订单。
2. 用户填写发票类型、抬头、税号、邮箱并提交。
3. 后台发票中心审核，驳回时记录原因，通过后进入待上传。
4. 管理员上传发票文件后状态变为已开票。
5. 小程序发票记录展示状态和发票文件。

## 六、当前字段缺口与待确认事项

| 编号 | 问题 | 影响模块 | 建议处理 |
|---|---|---|---|
| 1 | 订单编号在小程序使用 `orderNo`，后台使用 `id` | 订单、飞手分配订单、发票关联订单 | 后续接口统一命名，前端展示可仍叫订单号 |
| 2 | 小程序商品数据与后台商品数据当前不是同一份 mock | 商品、首页推荐、订单快照 | 后续以后台商品为唯一源，小程序只消费上架商品接口 |
| 3 | 首页轮播后台已配置，但小程序首页当前未直接消费 `state.media` | 小程序配置 | 小程序首页应接入轮播素材接口 |
| 4 | 飞行报备小程序表单字段少于后台详情字段 | 飞行报备 | 从飞手认证设备带出机型、序列号、UAS 码，并补充飞行区域、日期、时段、备注 |
| 5 | 培训报名小程序仍是通用预约表单 | 培训管理 | 需要独立课程字段：课程意向、联系人、手机号、备注 |
| 6 | 任务意愿取消、后台选定飞手后的状态未完全定义 | 任务需求 | 明确征集中可取消、关闭/选定后不可取消，以及是否生成派工单 |
| 7 | 飞手履约状态是否允许飞手主动更新仍未锁定 | 订单、飞手端 | 首版可只读；若开放完成订单，需要后台和用户端进度同步 |
| 8 | 运营商申请和意见反馈没有处理状态 | 运营管理 | 当前按信息收集处理；如要闭环需增加审核/处理/回复字段 |
| 9 | 地址簿没有后台独立管理 | 地址、订单 | 保持用户侧私有，后台只查看订单快照即可 |
| 10 | 关于我们电话、地址当前不结构化 | 关于我们、客服 | 建议后台拆出电话、地址、地图坐标字段，小程序直接消费 |

## 七、评审建议

1. 先确认商品、订单、支付、表单字段和派单规则，因为它们决定主交易链路。
2. 再确认飞手认证、飞手任务、分配订单和飞行报备，因为它们决定飞手端身份与履约闭环。
3. 然后确认发票、培训、运营商申请、意见反馈是否首版只做信息收集，还是要完整处理状态。
4. 最后确认首页素材、导航入口、关于我们和客服字段是否需要结构化配置。
